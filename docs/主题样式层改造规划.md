## 1. 改造目标

当前 `test` 分支的 SwiftUI 页面已经具备较完整的功能，但界面样式仍然主要分散在各个页面内部维护，表现为：

- 同类页面反复手写 `padding`、`cornerRadius`、`background`、`foregroundStyle`、`font`
- 相同语义的标题、说明文字、辅助文字，在不同页面使用了不同字级和颜色
- 卡片容器、表单容器、按钮样式存在重复实现
- 浅色/深色模式的视觉层级更多依赖系统默认效果，缺乏统一 token
- 后续继续做页面优化时，容易出现“局部调整好看，但全局风格越来越散”的问题

本次 Theme 改造的目标不是做一次“大重写”，而是建立一层可持续维护的 UI 基础设施：

1. 将颜色、字体、间距和通用组件外观集中管理。
2. 在不影响现有业务逻辑的前提下，逐步替换页面内分散的硬编码样式。
3. 先统一低风险、高重复、高收益的 SwiftUI 外观层，再逐步推进复杂页面。
4. 为后续 UI 优化、功能新增和设计调整提供稳定基础。

## 2. 本次改造的边界

### 2.1 本次要做的事

- 建立 `Theme` 基础目录与核心 token 文件
- 统一颜色语义、字体层级、间距层级
- 提供通用卡片、分组容器、主按钮、次按钮等 modifier / style
- 分阶段改造 `Stats`、`Browse`、`Settings`、`Decks`、`Review` 的 SwiftUI 外观层

### 2.2 本次不优先做的事

- 不先重构业务逻辑
- 不先重写 `ReviewSession`、`CardWebView`、同步流程、后端接口层
- 不先改 UIKit / WebView 细节页面的内部渲染逻辑
- 不把 Theme 改造和“功能修复”混在同一批次推进

### 2.3 风险控制原则

- **先 token，后组件，最后页面**
- **先低风险展示页，后高风险交互页**
- **单批次只改外观层，不顺手重写逻辑**
- **每一阶段都保证可回退、可验证**

## 6. Phase 1: 优先改造 Stats 模块

**当前状态**: 已完成

### 6.1 为什么 Stats 第一优先级最高

`Stats` 模块中存在大量结构高度相似的图表卡片：

- 统一的标题
- 统一的说明文字
- 统一的卡片容器
- 大量重复的 `.padding()`
- 大量重复的 `.background(.ultraThinMaterial, in: RoundedRectangle(cornerRadius: 12))`

这类页面最适合先接 Theme，因为：

1. 重复最多，Theme 收益最直接。
2. 逻辑风险低，主要是展示层。
3. 可以最快验证 Theme 的卡片、字体、颜色方案是否合理。

## 10. Phase 5: 改造 Review 模块的外围 SwiftUI 壳层

### 10.1 为什么 Review 不能太早动

`ReviewView` 是高复杂度交互页面，样式与功能耦合非常深：

- Show Answer 按钮紧凑模式
- 答题按钮布局模式
- 统计信息区
- 正确性提示与反馈信息
- 顶部工具栏与底部区域的显示逻辑

如果一开始就 Theme 化 Review，很容易把外观改造和交互逻辑改动混在一起。

### 10.2 建议优先改造范围

- `AnkiApp/Sources/Review/ReviewView.swift` 中的外围信息区
- 空态 / 完成态卡片容器
- 辅助说明文字
- 非核心按钮的字体、颜色、间距

### 10.3 本阶段暂不处理

- `CardWebView` 内部 HTML/CSS/JS 渲染逻辑
- 答题按钮行为逻辑
- typed answer / media / review state 逻辑

### 10.4 验收标准

- Review 外围 UI 风格与全局 Theme 对齐
- 不引入交互回归
- 不破坏答题体验和布局稳定性

## 13. 每阶段的验证点

每一阶段完成后都应至少检查：

1. 浅色 / 深色模式显示是否一致
2. Dynamic Type 下是否出现明显挤压或错位
3. iPhone 小屏幕下是否出现卡片容器溢出
4. 颜色语义是否一致
5. 是否引入非预期的点击区域或布局行为变化

对于高风险页面，还应补充：

- Review 流程手动验证
- Browse 编辑和保存流程验证
- Settings 页面导航与 Picker 显示验证

## 15. 最终验收标准

Theme 改造完成后，应达到以下结果：

1. 新页面不再需要从零决定标题字体、辅助文字、卡片背景和间距。
2. 同类页面之间的视觉差异明显减少。
3. 浅色 / 深色模式具有一致的层级和语义色。
4. 页面代码中的硬编码样式显著减少。
5. 后续 UI 优化能优先改 token / modifier，而不是逐页修补。
