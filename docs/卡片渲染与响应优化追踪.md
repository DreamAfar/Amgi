## 2. 卡片渲染空白：排查过程与根因演进

### 2.1 最早暴露出来的现象

- Review 页面存在“卡片内容空白 / 公式不显示 / 有时切换后才出现”的问题。
- 一开始看上去像是 MathJax 本身没生效，但实际不是单一根因。

### 2.2 第一轮判断：先修公式配置层

- 对应提交：e50e55c
- 这一轮先修了 `CardWebView` 中 MathJax 的 inline math 处理，让 `\\(...\\)` 这类行内公式能被正确识别。
- 同一轮还顺手修了统计页 review time 单位和 y 轴时间格式化问题，但对“卡片偶发空白”本身只能算缩小了范围，不是最终解。

### 2.3 第二轮判断：问题不只是公式语法，而是渲染时序

- 对应提交：3bdd746
- 后续排查发现，空白问题并不只是“公式 delimiters 配错”，而是 `CardWebView` 里公式 typeset、页面 startup、hook 执行、内容显隐之间存在时序竞争。
- 这也是为什么前一轮修完 inline math 后，症状仍可能以“空白 / 晚一点才出现 / 某些卡更明显”的形式残留。
- 当 loadHTMLString 用 baseURL: amgi-asset://card/ 时，这个回调被触发，amgi-asset scheme 不在白名单里，导致 decisionHandler(.cancel) 取消了整个导航 —— didFinish 从不触发，卡片永远空白。
- 根本原因：decidePolicyFor 导航策略只允许 、about:、javascript: 通过。当 loadHTMLString 使用 baseURL: amgi-asset://card/ 时，这个导航被识别为 amgi-asset:// scheme → 不在白名单 → decisionHandler(.cancel) 取消导航 → didFinish 永不触发 → pendingUpdateScript 永不执行 → 卡片空白。

  修复：在白名单加入 scheme == CardAssetPath.scheme（即 amgi-asset），让框架页加载正常完成。

### 2.4 第二轮真正收敛的改动

- 在 `CardWebView` 中把公式 typeset 与 startup/hook 的顺序稳定下来，减少页面先显示、公式后补、或 hook 先跑导致内容状态不一致的问题。
- 为暗色模式下的卡片增加背景与文字 fallback，避免卡模板 / CSS 没完整定义时，出现“内容其实在，但背景透明或前景色不对，看起来像空白”的情况。
- 同一提交还修了 DB check loading UX、Empty Cards 详情补全、Undo RPC method mapping，这些与渲染空白不是同一根因，但属于同一轮稳定性收口。

### 2.5 当前补充：再修一次 CardWebView 字符串转义隐患

- 当前工作区已修改：`CardWebView.swift` 中多行 Swift 字符串里嵌入的 JS 正则 `bg.match(/rgba?\((\d+),\s*(\d+),\s*(\d+)/i)`。
- 根因：Swift 多行字符串会把 `\(` 识别为字符串插值起点，导致编译器把 JS 正则内容误当成 Swift 表达式解析，引发 `d`、`\s`、`+` 一串级联报错。
- 处理方式：将该 JS 正则中的反斜杠全部按 Swift 多行字符串规则转义为双反斜杠，保证最终输出给 JS 的仍是正确正则。

## 3. CSS / 样式层在这轮问题中的作用

### 3.1 CSS 注入链路

- Review 卡片样式并不是单靠模板自身 CSS。
- 当前链路是：`ReviewSession` 侧把渲染结果的 CSS 注入 HTML，再由 `CardWebView` 的 frame HTML 包一层默认样式、CSS 变量和 runtime JS。
- 这意味着“看起来像内容空白”的问题，既可能来自模板本身，也可能来自外层默认样式、变量 fallback、颜色对比度或内容显隐时序。

### 3.2 这轮和 CSS 直接相关的几个落点

- 暗色卡片背景 fallback：避免模板没显式给背景色时，透明背景叠到宿主容器上，产生“像空白”的观感。
- Review 顶部工具栏与底部安全区跟随卡片背景色：让宿主 chrome 和卡片容器不再割裂。
- 资源链路优化里，对 Review 的媒体目录定位做了缓存，对资源文件读取改成 `mappedIfSafe`，属于 CSS/媒体资源加载环境层的优化，不改模板语义。

### 5.2 点击柱条导致列表无法滚动

- 对应提交：d95d189
- 根因不是图表本身，而是点击命中层用了 `DragGesture(minimumDistance: 0)`。
- 这个手势会抢走竖向滑动，导致手指落在柱条区域时，外层 `ScrollView` 失去滚动机会。
- 处理方式：统一改成 `SpatialTapGesture`，只保留点击命中，不再拦截竖向滚动。

## 6. 响应优化：从 Swift 层到桥接层

### 6.1 第一轮 Swift 侧减负

- 对应提交：43f5acf

#### 6.1 改动点

- `DeckClient` 新增 `fetchNamesOnly()`，Browse/Stats 的牌组筛选不再用带计数的 `getDeckTree()`。
- `BrowseView` 首次加载改为 deck/tag 并行，再做搜索。
- `NoteEditorView` 的 notetype 获取移到后台线程。
- `StatsDashboardView` 的 graphs fetch 放进 `Task.detached`，避免占主线程感知。

#### 结论

- 这轮主要是把“明明只是填筛选菜单，却顺手把整棵 deck tree counts 也算了”这类多余工作剥掉。

### 6.2 第二轮桥接层减 RPC 次数

- 对应提交：d0c54ab

#### 6.2 改动点

- Rust bridge 新增批量取 note 的 FFI 接口。
- Swift `NoteClient.fetchBatch()` 不再对每个 note 单独发一次 `GetNote` 调用。

#### 这轮的意义

- 浏览页一页 50 条笔记，之前是 50 次 bridge 往返；这一轮先把它压成一次 FFI 调用。

### 6.3 第三轮把“批量”从伪批量改成真批量

- 对应提交：af86535

#### 根因演进

- 上一轮虽然在 Swift 侧看起来是一笔批量调用，但 Rust bridge 内部仍是循环调用 `run_service_method(GetNote)`。
- 也就是说，减少的是 Swift ↔ C 往返次数，不是底层实际查询次数。

#### 最终处理

- 改成 Rust bridge 直接走一次 SQL，把请求的 note ids 一次性查出来，再重新打包成 note protobuf 数组返回。
- 这一轮才真正把“50 次查 note”收敛成“一次数据库查询 + 一次桥接返回”。

### 6.4 第四轮：首页 / 编辑页 / Review 资源链路再减负

- 对应提交：f3e8d67

#### 改动点

- App 首页进入时不再预热 Browse / Stats 首次加载，等用户真的切到 tab 再做。
- `NoteEditorView` 全量 tags 改成首次打开标签选择器时再加载。
- Review 资源链路里缓存当前用户 media 目录，并把资源文件读取改成 `mappedIfSafe`。
- 同一提交还包含 `CardWebView` 多行字符串里 JS 正则转义修复。

## 7. 当前工作区未提交：启动链路优化

### 当前思路

- 如果本地已经有 deck tree / 热力图快照，则不再等 `openCollection()` 完成后才进入主界面。
- 先进入 UI，再后台开库；开完后通过 `AppCollectionEvents.didOpenNotification` 通知 Decks / Browse / Stats 拉真数据。
- 如果本地没有快照，仍走原来的阻塞启动，避免首次安装或空账号出现“先进入 UI 但没有任何可显示内容”。

### 目前已改到的文件

- `AnkiApp/Sources/AnkiAppApp.swift`
- `AnkiApp/Sources/Shared/AppCollectionEvents.swift`
- `AnkiApp/Sources/Decks/DeckListView.swift`
- `AnkiApp/Sources/Browse/BrowseView.swift`
- `AnkiApp/Sources/Stats/StatsDashboardView.swift`
- `AnkiApp/Sources/ContentView.swift`

### 设计意图

- 把“启动体感”从“等库开完再见首页”改成“先见首页缓存，再等真库接管”。
- 这个改动更偏“可交互时机优化”，不是减少 `openCollection()` 本身的耗时。

## 9. 当前工作区补充：Review / Stats / Browse 收口修复

### 9.1 Review 渲染与交互

- `CardWebView` 里不再把所有脚本节点一律重写执行，而是只对真正的 JS/module 脚本做 deferred replace，保留 MathJax 的 `math/tex` 数据脚本，修复公式不渲染的问题。
- 模板中的 `style` 和 `link rel="stylesheet"` 现在会在卡片 HTML 注入时同步挂到 `document.head`，解决模板正反面外链 CSS 引用不生效的问题。
- Image Occlusion 增加文本遮罩的测量、绘制和命中测试，修复“切换遮罩对文字遮罩不生效”。
- `ReviewSession.refreshAfterCardMutation()` 改为保留当前正反面状态，不再在模板/笔记类型编辑后强制切到答案面。
- `ReviewView` 增加“工具栏背景自动匹配卡片”的可控设置，同时保持底部回答按钮容器透明，避免再次遮挡卡片内容。

### 9.2 Stats 图表

- `FutureDueChart` 将 backlog 折叠到当前窗口左边界并显式锁定 x 轴 domain，减少 backlog 模式下的等待和两侧留白。
- `EaseChart` 对齐上游 difficulty 百分比语义，修正平均值与刻度显示，并为 FSRS difficulty 使用绿 → 黄 → 红渐变。
- `IntervalsChart` 显式传入 `xMax` 并设置 x 轴 domain，避免稳定期/间隔图在两端留空。
- `StatsDashboardView` 新增图表排序按钮、拖拽排序 sheet 和持久化顺序。
- `CardCountsChart` 新增“分开统计暂停/搁置的卡片”开关，默认保持当前分开展示。

### 9.3 Browse / Empty Cards

- `EmptyCardsView` 的每条空卡片笔记现在可以直接点开 `NoteEditorView`，修改后自动刷新空卡报告。
- Browse 顶部牌组/标签筛选支持再次点击取消、当前牌组下动态标签、长按“全部标签”展开整页标签选择，并自动滚动到当前筛选项。
- Browse 排序从旧的本地 `BrowseSortMode` 切到后端 `SearchRequest.order.builtin`，支持：到期、卡片修改日期、卡片模板、复习、标签、添加日期、牌组、笔记修改日期、笔记模板、简易度、遗忘、间隔，以及顺序/倒序。
- Browse 列表项新增笔记模板副标题，并提供开关；关闭后会直接跳过模板名称映射加载，避免任何额外负担。
